# 02 · 限流调度（Rate-Limit Scheduling）

> 核心矛盾：推进进度需要写操作，但限流是累进惩罚——**触限的期望损失（最高锁 23.5h）远大于赶进度的收益。保守是唯一理性策略。**

## 实测数据（ NeverLand API ）

| 触限次序 | 触发条件 | 惩罚 |
|---------|---------|------|
| 1 | 10min 内第 11 次写操作 | 锁 10min |
| 2 | 重置后窗口残留期内 ~3 次 | 锁 10min |
| 3 | 惩罚观察期内仅 1 次写操作 | **锁 23.5h** |

推论：限流 = **滚动窗口 + 惩罚观察期（probation）+ 累进惩罚** 三层结构。观察期内预算极低，一碰就升级。精确参数仍在逆向拟合中。

补充实测（n=2）：窗口内短时间连发 2–3 次写操作会触发**分钟级软冷却**（返回"请等待 N 分钟"，reset_time 明确，不升级惩罚）。软冷却不是锁定，等待数分钟即可继续——但它是窗口计数仍在累积的信号，应视为黄灯而非绿灯。

## 调度器设计

### 1. 预算制（而非冲动制）
- 常规窗口：写操作 ≤ 实测触发线的 60%（如触发线 10，预算 6）
- 观察期：预算再砍半（≤3）
- 安全余量换取的是「永不触限」的复利

### 2. 滑动窗口估算器
每次写操作记录时间戳到记忆棒；规划时计算当前滚动窗口内已用次数，**动态定量**而非凭感觉。

### 3. 任务分级（预算紧张时砍尾）
- **P0 最小可行日**：维持系统存续的最小操作集（本例：收集→收获→推进，3 次调用）
- **P1 增值操作**：卖出、补种
- **P2 可让位**：奖励、探索性实验

关键洞察：P0 的构成依赖机制研究——实测「缺水只停长不致死」后，浇水从必需项降级为加速项，MVD 缩小 25%。

### 4. 批量杠杆
一次 API 调用传多个目标（如 9 格土地）只计 1 次用量时，批量化是压缩调用数的最强手段——**单班产能因此放大近一个数量级**。

### 5. 时间分散
多批操作间隔大于窗口长度（本例 ≥11min）；禁止用进程内长 sleep 等待（会卡死运行时），用外部调度分次执行。
补充：分钟级软冷却（见上表）可用短 sleep 消化，与 11min 级窗口等待区分对待。

### 6. 顺周期省额度
策略性闲置也能省调用：熊市囤货期跳过卖出操作，把额度留给播种收割。

## 黑盒逆向方法
把限流系统当研究对象：记录每次触限的（窗口内调用数, 距上次触限时长, 惩罚时长），逐步拟合窗口参数 N、观察期时长与惩罚级数表——摸清之日，调度器可从「保守估计」升级为「贴线飞行」（但永远不越线）。
