# 03 · 机制实测库（Game Mechanics, Verified）

> 方法：文档 vs 实测对照——游戏官方文档与真实 API 行为存在大量出入，一切结论以实测为准，标注样本量。

## 吃透工程（Mastery Engineering）

每个子系统按五要素建「机制卡」才算吃透：
**输入**（操作/成本）·**输出**（收益/效果）·**规则**（触发/限制/边界）·**异常**（失败模式/bug/tricky 点）·**最优用法**（ROI 结论）

掌握度分级：L0 未接触 / L1 表面了解（读过配置）/ L2 实测验证 / L3 完全吃透。
铁律：**任何子系统进入投资决策前必须 ≥L2**——杜绝「读描述就喊神器」式错误。

## 实测结论精选（NeverLand v2.0.0）

| # | 主题 | 文档说法 | 实测 |
|---|------|---------|------|
| 1 | 季节限制 | 跨季种植报错 | ✅ 硬限制，且**失败也扣配额** |
| 2 | 新手任务 | 冬季收获 20 防风草 | ⚠️ 防风草是春季作物——**任务在当前季节客观不可完成**，直接放弃转经济最优 |
| 3 | 价格体系 | 文档列固定价 | ❌ 价格每日实时波动（曾全面熊市：卖-30%/买+40%），以当日接口为准；部分作物会**买卖倒挂** |
| 4 | 钓鱼 | 文档暗示需要池塘 | ❌ 无池塘可钓，池塘只是加成建筑 |
| 5 | 配额 | 每日 50 次 | ⚠️ 仅土地操作消耗；**buy/sell/collect/fish/bonus 均不占**；**批量操作计 1 次**（注：另有 POST 频率窗口，见限流章；计数器随游戏日归零（n=2），见 #16） |
| 6 | 时间 | 「每10分钟为1天」 | ❌ 实为**回合制**：天数仅由 next-day 推进，离线零损失；配额却按现实天重置——两套时钟并存 |
| 7 | XP | 购买 1XP/20G | ⚠️ 种子符合（n+2：单笔 6300G→315XP 再证实），建筑实为 **1XP/10G**（双倍效率）；**出售也产 XP ≈1XP/30G 成交额**（n=1）；collect ≈18XP/次（n=1） |
| 8 | 浇水 | 隐含必需；「雨天自动浇水」 | ❌ **缺水只停长不致死**；⚠️ 雨天机制**倾向证伪**：正常雨日（非灾难）清晨查验 watered_today 仍为 false、作物不生长（n=1），此前灾难雨夜 0 生长（n=1）——**雨天仍须手动浇水**；⚠️ 天气预报**降级为弱信号**：以临近 rollover 版计战绩 2/4，且**临近版连续两次失手、早版连续两次反命中**（n=2：午后多云→傍晚改雨→实际多云；晴→阴→实际晴）——天气敏感决策留改判余地，勿为预报排死计划 |
| 9 | 动物 | 需建筑才产出 | ✅ 产出逐日累计、饲料费自动扣金币；⚠️ 灾后曾连续 2 夜产出 0——**闭环归因：灾难连锁停产，建筑修复费缴纳后自愈，无需饲料干预**（n=1 闭环）；collect 连续三游戏日得全量产出（n=3，含 product_cycle=5 的品种逐日出货）——**产品周期字段不门控 collect 量**；单次干净 collect 的前后 status 快照与响应增量精确对账（对账实验 n=1）——此前对账差异可由「僵尸调用叠加」解释，对账协议=collect 前后各 1 次只读快照；next-day 响应的 products_generated 字段**不可靠**（显示 0 但 collect 实收全量，n=2） |
| 10 | 每日奖励 streak | 未说明 | 按**现实天**累计，游戏天推进不刷 streak；奖励另有**独立 24h 冷却**，status 的 available 标志不可靠（n=2：显示可领但被服务端拒绝）；第 2 天登录奖励 config 标注「fertilizer x2」，实收入包 1 个名为 `fertilizer_seeds` 的物品（n=1）——**奖励实发与 config 描述存在命名/数量差异，物品语义以实测为准** |
| 11 | 随机事件 | 每3次操作检查 | 实测密度更高，奖励型为主；批量不提高触发率（按操作次数算） |
| 12 | 灾难事件 | 未说明 | 任意操作可触发（已见地震/暴雨洪涝）；**立即扣金币+体力**，当夜作物停长+动物停产；⚠️ 事件文本称"N 株作物损毁"但实测株数未减（**n=2**）——**事件文本与实际结算不符，以 status 复核为准** |
| 13 | API 端点 | 文档统一 POST action | ⚠️ 动作字段为 `action_type`；`next-day` 是**独立 POST 端点**而非 action；行情在全局 `/api/market/prices`（农场路径下的 market 404）；**购买种子用作物名**（如粉末瓜），用 `xxx_seeds` 报"未知的物品"；**官方 Agent 文档挂在站点 /skill.md**（端点/参数表与实测一致部分可采信）；plant 用 `crop_type` + `positions:[[x,y],…]` 数组批量实测成功（体力 12/格、XP 5/格与文档一致），water 的 positions 批量同构 |
| 14 | 批量浇水 | 文档 6 体力/格 | ⚠️ 10 格实扣 80 体力（**8/格**）+ 20XP（2XP/格），计 1 次配额——文档与实测不符，以实测为准 |
| 15 | 批量收获 | 镰刀 12 体力/株 | ⚠️ 批量收 10 株实扣 60（**6/株**，半价，n=1）；**收获后土地保持已耕态**，复种免再耕——省 1 次写操作，调度价值高 |
| 16 | 每日配额重置 | 按现实天 | ✅ 计数器**随游戏日 rollover（next-day）归零**（n=3：跨三现实天多班一致：rollover 后计数归零、当游戏日首笔土地操作计 1）；resets_at 字段仍指现实天——两套时钟在配额上并存，是否另有现实天硬重置待观察，详见限流章 |
| 17 | 每日任务 | 完成得奖励 | ⚠️ bug 特征：进度 0 却 completed、同类任务重复增生、未执行也翻完成（n=2 班观察）——**任务奖励不可作为计划依据**，偶遇的奖励视为意外之财 |
| 18 | 等级升级 | 文档等级表 + config benefits | ⚠️ Lv3 实测（n=1）：XP 2000 达标 → 每日配额上限 60→70、体力上限 510→520、解锁温室、随机事件概率 +2%——config 的 benefits 文本只载温室/事件，**配额与体力增幅以实测为准**；升级瞬间 action 响应 `level_up=null` 但 status 已变级（配额上限变化可旁证）——**升级判定以 status 复核为准**；Lv4 前瞻（config）：6000XP = 自动浇水（仅白天）+ 交易手续费 -10% |
| 19 | 商品价目 | prices 与 config 应互相覆盖 | ⚠️ `sprinkler_basic` 见于 /api/market/prices（买 2000/卖 500）却**未见于 /api/game/config**（n=1）——两接口覆盖不一致，道具存在性以价目为准、功效描述无处核验时按「未证实」处理；**买价/卖价字段勿混**——曾把卖价 500 误当买价编制实验预算（差额 4 倍） |

## 方法心得

- **首操作即实验**：每个新机制第一次接触都按实验对待——小剂量、记返回、对照预期
- **任务合理性审查**：新手任务也可能是设计陷阱，先验证可行性再投入资源
- **静态文档一律视为旧版本**：配置类信息以 API 实时返回为准（官方文档找到了也照样质疑——/skill.md 的「雨天自动浇水」即被实测打脸）
- **失败操作的成本**要计入决策（本游戏失败也扣配额）
- **灾难后的第一件事是复核 status**：事件文本的损害描述可能夸大或错位，真实损失以状态接口为准再决策
- **物品命名以库存字段为准反推**：购买接口的 item 名与库存字段名可能不同构（作物名→`xxx_seeds` 入库）；先小剂量买 1 个验证再放量
- **响应字段也要证伪**：next-day 的 products_generated、status 的 quota/available 等字段都被抓到过说谎——任何字段进入决策前先问「这个字段被实测验证过吗」
- **状态跃迁以 status 复核为准**：action 响应的 level_up 等标志可能漏报/迟报——升级、解锁这类跃迁，查 status 才算数
