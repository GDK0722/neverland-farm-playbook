# 03 · 机制实测库（Game Mechanics, Verified）

> 方法：文档 vs 实测对照——游戏官方文档与真实 API 行为存在大量出入，一切结论以实测为准，标注样本量。

## 吃透工程（Mastery Engineering）

每个子系统按五要素建「机制卡」才算吃透：
**输入**（操作/成本）·**输出**（收益/效果）·**规则**（触发/限制/边界）·**异常**（失败模式/bug/tricky 点）·**最优用法**（ROI 结论）

掌握度分级：L0 未接触 / L1 表面了解（读过配置）/ L2 实测验证 / L3 完全吃透。
铁律：**任何子系统进入投资决策前必须 ≥L2**——杜绝「读描述就喊神器」式错误。

## 实测结论精选（NeverLand v2.0.0）

| # | 主题 | 文档说法 | 实测 |
|---|------|---------|------|
| 1 | 季节限制 | 跨季种植报错 | ✅ 硬限制，且**失败也扣配额** |
| 2 | 新手任务 | 冬季收获 20 防风草 | ⚠️ 防风草是春季作物——**任务在当前季节客观不可完成**，直接放弃转经济最优 |
| 3 | 价格体系 | 文档列固定价 | ❌ 价格每日实时波动（曾全面熊市：卖-30%/买+40%），以当日接口为准；部分作物会**买卖倒挂**；**熊市可品种脱钩**（某瓜类第 4 天回升至 -4.5% 而全市场仍 -30%，n=1）；**行情按现实日刷新**（同一现实天跨两个游戏日，价格逐分不差；prices 的 date 字段=现实日期，n=1）；⚠️ 但**同一现实天内报价日内可变**（某瓜类卖价半日内 862→840，n=1）——「逐分不差」口径修正为「游戏日间不变、现实天内可能变」，大宗出货前以卖出前最新报价为准——价格序列按现实天索引，同现实天内「等下个游戏日的好价」是空等；**成交价≠显示价**（卖出实价 vs 显示价：-6.5% 与 **0.0%** 两样本——**隐性手续费假说动摇**，偏差更似行情快照时滞/噪声而非固定费率，Lv4「手续费 -10%」解锁后复测可终审，n=2；买入实价 vs 显示价：+0.9%/0.0%/0.0% 三样本 ≈ 零偏差，n=3）——大宗交易按显示价打折估算、成交后按响应实额回填；⚠️ prices 的 season 字段与农场季节不一致且会漂移（春→夏而农场仍为冬，n=2，市场历法独立或 bug） |
| 4 | 钓鱼 | 文档暗示需要池塘 | ❌ 无池塘可钓，池塘只是加成建筑；**动作级独立冷却 = 600s 石锤**（reset_time 拟合 n=2 + 边界验证 +340s 拒 / +646s 过，方法见限流章「冷却探针法」）——fish「不占每日配额」成立且再证实（冷却期间 `daily_quota.used` 不动；⚠️ action 响应的 `quota_used` 字段实为**当日累计口径**，非本操作消耗）；**钓鱼不产 XP**（clean n=2：state_changes 无 xp 字段 + 前后快照 XP 不变；早年响应丢失样本的 +73 归档「构成不明」）；体力 -15/次稳定；鱼获估值见 action 文本 |
| 5 | 配额 | 每日 50 次 | ⚠️ 仅土地操作消耗；**buy/sell/collect/fish/bonus 均不占**；**批量操作计 1 次**（注：另有 POST 频率窗口，见限流章；计数器随游戏日归零（n=6），见 #16） |
| 6 | 时间 | 「每10分钟为1天」 | ❌ 实为**回合制**：天数仅由 next-day 推进，离线零损失；配额却按现实天重置——两套时钟并存；**体力随 next-day 回满**（next-day 响应 energy_restored 字段+status 复核，多班一致）——rollover 前余力不用即浪费 |
| 7 | XP | 购买 1XP/20G | ⚠️ 种子符合（n+4：单笔 3500G→175XP 再证实），建筑实为 **1XP/10G**（双倍效率）；**特殊物品购买无 XP**（500G→+0，n=1）——购买 XP 规则三分；**出售也产 XP ≈1XP/30G 成交额**（n+1：实额大宗卖出精确命中）；collect ≈18XP/次（n=4） |
| 8 | 浇水 | 隐含必需；「雨天自动浇水」 | ❌ **缺水只停长不致死**；⚠️ 雨天机制**倾向证伪**：正常雨日（非灾难）清晨查验 watered_today 仍为 false、作物不生长（n=1）——**雨天仍须手动浇水**；⚠️ 天气预报**降级为弱信号**：战绩 3/9（临近版三连失后首次命中、此后又连失：rainy 0.85→sunny、cloudy 0.85→sunny ×2），且预报当日内可变（n=4：同现实天内 cloudy 0.85 与 rainy 0.85 先后改报又一例；⚠️ 最强样本——同一班次两次 GET 间隔仅 21 秒，预报 sunny 0.85→rainy 0.85 秒级翻转，疑似按请求重掷）——天气敏感决策留改判余地，勿为预报排死计划 |
| 9 | 动物 | 需建筑才产出 | ✅ 产出逐日累计、饲料费自动扣金币；⚠️ 灾后曾连续 2 夜产出 0——**闭环归因：灾难连锁停产，建筑修复费缴纳后自愈，无需饲料干预**（n=1 闭环）；collect 连续四游戏日得全量产出（n=4，含 product_cycle=5 的品种逐日出货）——**产品周期字段不门控 collect 量**；**未收集产出不跨日累积**（n=1：故意跳过当日 collect，次日对账仅得单日量，前漏收确认损失——**动物产出当日必收**）；单次干净 collect 的前后 status 快照与响应增量精确对账（n=4）；next-day 响应的 products_generated 字段**不可靠**（显示 0 但 collect 实收全量，n=4）；⚠️ 饲料代扣金额异常**结案**：某次 rollover 代扣达配置饲料费合计的 11 倍，此后连续两夜 rollover 代扣为 0——判为**一次性扣费**（n=1 鉴别+n=2 无复发），成分不明归档不再追踪；**灾害机制分叉**：龙卷风当夜动物**不停产**（n=1，与地震停产相反，详见 #12） |
| 10 | 每日奖励 streak | 未说明 | ✅ **按现实天累计（每现实天首个 next-day 触发）——石锤**：同现实天双 next-day 分离实验，第二个 next-day 后 streak 不变（3→3），「按 next-day 次数」假说证伪；此前「rollover 后跳变」反例归因完成——现实天边界与 next-day 天然共线的观测假象（实验章判读陷阱案例闭环）；奖励另有**独立 24h 冷却**，status 的 available 标志不可靠（n=4：显示可领但被服务端拒绝；又一例——POST 日锁期内 available 仍显示 true，而任何 POST 必被 429，字段与锁状态完全脱节）；⚠️ 拒绝响应的 `reset_time` 是**未来时刻**（下次可领点），交接记录勿写成「冷却早过」这类歧义句——曾因此白试一发（损失仅一次窗口计数，但属可避免的误读）；第 2 天登录奖励 config 标注「fertilizer x2」，实收入包 1 个名为 `fertilizer_seeds` 的物品（n=1）——**奖励实发与 config 描述存在命名/数量差异，物品语义以实测为准**；**streak 与奖励领取解耦（n=1）**：claim 被 429 拒绝的当天，status daily_streak 照常累计未断——streak 由当日首个 next-day 触发计数、与奖金领取解耦，429 损失仅限当日奖励本身；交接记录勿凭推断写「streak 已断」，以 status 复核为准 |
| 11 | 随机事件 | 每3次操作检查 | 实测密度更高，奖励型为主；批量不提高触发率（按操作次数算）；⚠️ Lv3（事件率+2%）后体感暴增：**连续 3 次购买中 2 次触发灾难**（n=1 班）——高等级每次 POST 都是事件抽奖；n+2 班：7 次 POST 触发 2 事件（奖励 +36XP／惩罚虫害）与 7 次 POST 触发 2 事件（xp_bonus +71XP／seed_bonus 白送种子）≈29%——与文档「每 3 次操作一次检查」口径相容；事件可由 sell/plant/collect/buy 等各类操作触发；**新类型**：seed_bonus（plant 触发，白送种子——注意可能送出非当季品种，跨季不可种）；xp_bonus 数值不稳（+36/+71/+86 三样本，上行趋势待证） |
| 12 | 灾难事件 | 未说明 | 任意操作可触发（已见地震/暴雨洪涝/盗贼工会/龙卷风）；**立即扣金币+体力**；⚠️ **事件文本与实际结算系统性不符（n=4）**：报损株数未减（虚报 11 株超过实有 10 株；雷暴「击毁 7 株」而 15/15 在田）、"物品损失"未结算；但**比例扣金（-15%）与体力扣除是实的**——**事件文本只当气氛组，损失以 status 复核为准**；**雷暴样本新分叉（n=1）**：金币扣除与 config 区间吻合、报损株数虚报、体力实扣却**远超**文本值（≈109 vs 文本 39，config 区间 30-50）、另出现来源不明的 XP 与种子进账——灾害结算可能是多路独立掷骰，文本只报了其中一路；**对照 config 的 effect 区间可判断哪些文本数字是真实掷骰、哪些是虚报**；**灾害机制分叉（n=1 each）**：龙卷风=仅扣体力+文本虚报，**当夜作物不停长、动物不停产**；地震=扣金+体力+当夜停长+动物停产——**龙卷风夜周期照常推进，地震夜须补救重排**（按灾种分级应对，勿一刀切） |
| 13 | API 端点 | 文档统一 POST action | ⚠️ 动作字段为 `action_type`；`next-day` 是**独立 POST 端点**而非 action；行情在全局 `/api/market/prices`（农场路径下的 market 404）；**购买种子用作物名**（如粉末瓜），用 `xxx_seeds` 报"未知的物品"；**官方 Agent 文档挂在站点 /skill.md**；plant 用 `crop_type` + `positions:[[x,y],…]` 数组批量实测成功（体力 12/格、XP 5/格与文档一致，n+1），water 的 positions 批量同构，**water 亦支持 `mode:"all"` 免坐标**（n+1）；**坐标系实锤 x=列、y=行**（种植返回的 position_x/y 与网格对照，n=1） |
| 14 | 批量浇水 | 文档 6 体力/格 | ⚠️ 实扣 **8/格**（10 格=80、15 格=120，n=2）+ 2XP/格，计 1 次配额——文档与实测不符，以实测为准 |

## 方法心得

- **首操作即实验**：每个新机制第一次接触都按实验对待——小剂量、记返回、对照预期
- **任务合理性审查**：新手任务也可能是设计陷阱，先验证可行性再投入资源
- **静态文档一律视为旧版本**：配置类信息以 API 实时返回为准（官方文档找到了也照样质疑——/skill.md 的「雨天自动浇水」即被实测打脸）
- **失败操作的成本**要计入决策（本游戏失败也扣配额）
- **灾难后的第一件事是复核 status**：事件文本的损害描述可能夸大或错位，真实损失以状态接口为准再决策
- **物品命名以库存字段为准反推**：购买接口的 item 名与库存字段名可能不同构（作物名→`xxx_seeds` 入库）；先小剂量买 1 个验证再放量
- **响应字段也要证伪**：next-day 的 products_generated、status 的 quota/available 等字段都被抓到过说谎——任何字段进入决策前先问「这个字段被实测验证过吗」
- **状态跃迁以 status 复核为准**：action 响应的 level_up 等标志可能漏报/迟报——升级、解锁这类跃迁，查 status 才算数
- **检索结构先于检索词**：config 类接口是列表还是字典、键名是什么，先 dump 结构再查找——一次"未见于 config"的误判差点冻结整条实验线
- **回溯验证法**：新模型提出后，用历史日志逐夜回代——全部吻合才升级为结论（生长阶段模型即以此法 n=3 证实）
- **逐项核算揪残差**：XP/体力/金币每班逐项对账——残差=未知机制的信号（曾借此发现被动回复疑似存在）；零残差说明无干扰时完全可预测
- **显示价与成交价都要记**：价格双轨发现于一次大宗卖出——报价接口和成交响应各记一笔，差额本身就是新机制（费率/点差）的信号
- **冷却探针法**：新动作首用后按梯度间隔（如 +5/+8min）试探第二次，被拒时读响应 `reset_time` 反推精确冷却——两三点即可拟合 CD（fish 600s 即如此定案），冷却类拒绝不升级惩罚，是零成本侦察
- **僵尸命令核验协议**：被运行时杀死的命令，其中 POST 一律按已发出计入窗口日志；用只读接口逐项核验资源/库存/计数器后再决策；响应丢失的样本标注「构成不明」，严禁编造回填——一虚惊一实锤两度实战，协议有效
- **导师先验优先采信**：有经验者一句「那东西我用过没成功」，价值等于整轮实验预算——两轮布防实验最终归档的结论与先验一致；先请教再设计实验（见实验章）
