# 01 · 循环工程（Loop Engineering）

> 核心问题：定时唤醒的 Agent 每班都是全新实例、上下文清零，如何像有记忆一样连续运营？

## OODA 班次循环

```
OBSERVE 感知 → ORIENT 研判 → ACT 行动 → LEARN 复盘 → （记忆回写）
     ↑                                                    │
     └──────────────── 记忆棒交接 ────────────────────────┘
```

- **OBSERVE**：先读记忆棒（上一班交接的结构化状态），再拉取环境实时状态，对齐差距
- **ORIENT**：差距分析 → 本班计划 → 每个操作预写预期结果
- **ACT**：批量执行，每步记录实际返回；异常立即归因
- **LEARN**：预期 vs 实际逐条对照，偏差强制归因（外部环境 / 内部决策），更新记忆棒

要点：**预写预期**是防呆关键——没有预期的执行，错误会静默累积。

## 记忆棒模式（Baton Pattern）

跨班交接不依赖对话历史，而依赖一份结构化文件（state.json）：
- 状态快照（资源/进度/在田资产）
- KPI 与研究队列
- 下一班计划草案
- 遗留问题与导师洞察

班次开工读它、收班重写它——**热启动，不考古**。

## 跨会话持久化的三次架构演进（教训实录）

| 代际 | 方案 | 结局 | 教训 |
|------|------|------|------|
| v1 | 平台文件系统 `/mnt/agents` | ❌ 定时实例是独立沙箱，文件不可见 | 平台的「持久存储」可能只在同会话内持久 |
| v2 | 用户上传附件 + 会话历史 | ❌ 触发实例读不到附件与历史 | 任何依赖「另一个实例留下痕迹」的通道都需实证 |
| v3 | cron content 内嵌 | ⚠️ 可靠但有长度天花板 | 适合保底知识，不适合完整知识库 |
| v4 | **外部 Git 存储（GitHub repo）** | ✅ 当前架构 | 公网存储是所有沙箱的公约数；raw 读取零认证，push 回写带历史 |

**元教训：跨实例的一切，必须随信使同行或存于中立驿站，不能指望平台内部通道。**

## 异常分级自愈

| 级别 | 情形 | 策略 |
|------|------|------|
| L1 | 单次操作失败 | 归因后当班调整，继续 |
| L2 | 频率锁定等资源限制 | 停手，计划移交下一班（游戏若为回合制则零损失） |
| L3 | 连续 2 班无法推进 | 深度诊断：重读配置、小步试探 |
| L4 | 账号事故/规则剧变/L3 无解 | 请求人类介入（**唯一允许的打扰**） |

配套纪律：L4 上报时必须零写操作——在基础设施不可信时蛮干等于伪造数据，比停手更糟。

## 回写双通道

状态回写采用降级链：外部存储 push → 调度器 content 更新 → 会话文本留痕。任一通道存活，记忆不丢。
